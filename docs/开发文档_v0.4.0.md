## 2. v0.4.0 关键变化（相对 v0.3.0）：持续迭代修订

### 2.1 修订机制
- 每个产出（实录/纲目/撷要/笺注/脉络）下方常驻**修订框**。填入意见 → 基于「该产出**当前版本内容** + 修订意见」再生成完整产出 → 存为**新版本**，版本 note 即修订意见。
- 设计取舍：修订只看「当前产出 + 意见」，不重新从原始素材推导（贴合"这段展开/术语表补 X/语气更正式"这类编辑型诉求）。日后如需"参考原始逐字稿再改"可扩展输入范围。
- 安全红线沿用：不改动数字/金额/日期/人名/公司名（实测「8.5%」等被保留）。脉络图修订特例：仍输出 `flowchart LR` Mermaid。

### 2.2 版本历史
- 存储层早已支持版本（`write_artifact` 每次写 +1 版本并存 note，`list_versions`/`read_version`）；本版把它接到前端。
- **版本切换器**：≥2 版本时出现 v1/v2/…（当前版高亮）；可查看任意历史版本（只读 + 显示当时修订意见）。
- **恢复历史版本**：查看旧版本时「恢复此版本为当前」——把旧版内容写成一个新版本，历史不抹除。

### 2.3 复用既有后台通道
- `runner.py` 抽出通用 `_start(sid, step, events_factory)`；`start_step`（生成）与 `start_revise`（修订）共用，仍以 `(sid, step)` 为并发维度——同一产出同一时间只允许一个任务。
- 修订进度复用 `GET /run-step-stream?step=X`（SSE），前端复用 `attachStream`。

## 4. 修订相关 API

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/sessions/{id}/revise-step?step=X` | 启动单步修订（body `{instruction}`；幂等；进度复用 run-step-stream）|
| GET | `/api/sessions/{id}/artifacts/{name}/versions/{v}` | 读某历史版本内容 |
| POST | `/api/sessions/{id}/artifacts/{name}/versions/{v}/restore` | 恢复某历史版本为当前（写新版本）|

（其余 API 见 [开发文档_v0.3.0.md](开发文档_v0.3.0.md) §4。）
