## 2026-06-15 · v0.5.0：实录分发言人 + 纲目连续化 + 徽标随进度

### 诉求
- 实录：先要求去发言人/时间戳，后修正为**保留并区分发言人**——不同发言人明显区隔，但同一发言人连续发言不被切断、保持完整；时间戳仍去除。
- 纲目：阶段出现「1–6 又从 1–5」的碎片化（分段各自重新计数）；且实录去时间戳后，阶段时间需改从上传的原始转写稿对位。
- 新功能规划：加「提问」窗口，AI 据素材 + 产出答疑——由我安排排期。
- 用户发现右上角徽标还固定显示「P1」，应随进度走到 P2。

### 做法
- 实录切分维度从「长度/时间桶」换成「**说话人轮次**」：`parse_turns`（同一说话人连续行合并）+ 重写 `build_correction_units`（永不切断一个轮次，超长才按句切分标 `is_continuation`）；`_match_speaker` 兼容多种说话人格式并规避裸角色词误判。提示词改为分发言人加粗排版、连续发言并段。
- 纲目长稿走 **map-reduce**（`_make_chapters`：先分段提炼脉络、再单次汇编连续阶段）；时间靠新增 `build_timeline_digest`（逐行从原始稿取时间锚点，行首/说话人标签后都识别）对位标注。
- 徽标：`/health` 返回 `phase=P2 + version`，`config.APP_VERSION` 读根 `VERSION` 作单一版本源，`FastAPI(version=…)` 同源；前端徽标显示 `v0.5.0 · P2`。
- P5 智能问答排在 **P3 辅助素材之后**（问答要「据资料回答」，PDF/图片素材需先由 P3 接入）。

## 2026-06-16 · v0.6.0：P3 辅助素材（一）课堂笔记照片视觉识别

### 起点与排序
- 重新审视进度，下一步在 P3 辅助素材 / 导出 / P5 问答 / 工程小项里，用户选 **P3 辅助素材**。
- 我原排"PDF 抽文本（低风险）先行、图片视觉后做"；用户指出真实高频场景是「**转写 + 课堂笔记照片**」，建议**先做图片识别、PDF 顺延**。采纳——图片视觉正是当前"半成品中的半成品"，价值更高。

### 设计对齐（3 个决策）
1. **笔记照片 = 辅助素材**：转录文字注入 **撷要/笺注**（知识归纳），**不混入实录**（实录是讲话逐字稿）。与前端既有文案一致。
   - 我把原方案里的"也进纲目"**收回**：纲目是按原文时间戳对位的时间轴骨架，照片无时间戳，强行注入会污染时间对位；已在交付时向用户说明，可后续按需接入。
2. **自动选视觉模型** + **用户可手动指定**：用户额外要求"在模型配置界面手动选图片识别模型"。→ 面板底部加全局「图片识别模型」选择器（供应商+模型，或留自动）。
3. **转录时机**：生成撷要/笺注时按需转录，按「文件名+mtime」缓存到 `derived/notes/`，重复生成不重复调用。

### 实现
- **Provider 多模态**：`Message` 加可选 `images`（`ImagePart` base64）；Anthropic image block / OpenAI image_url 两套 `chat()` 带图时拼多模态，无图行为不变。
- **`pipeline/vision.py`**：`resolve_vision_provider`（手动优先→自动选支持视觉的→无）、`transcribe_image`（Pillow 降采样到 1568px，缺失时原图兜底）、`collect_note_text`（缓存 + 并行转录 + 统计）。
- **store**：顶层 `vision_id/vision_model` + `get_vision/set_vision`（老文件缺字段=自动）；API `GET /providers` 返回 vision、`PUT /vision-model` 设置。
- **engine**：撷要/笺注收集笔记并注入；进度提示"识别 N 张 / 复用缓存 / 失败 / 无视觉模型"。
- **prompts**：`note_transcribe_*` 转录提示词（公式 LaTeX、表格 Markdown、字迹不清标记、不杜撰、数字人名照原样）；`minutes_user` 加可选笔记块（辅助定位、冲突以逐字稿为准）。
- **前端**：ModelConfig 底部图片识别模型选择器；Workbench 图片行 📷 标识、accept 加 webp、文案更新。
- 依赖加 `Pillow>=10.0`。
