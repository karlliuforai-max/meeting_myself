## 2. v0.5.2 关键变化（相对 v0.5.1）：纲目解除实录依赖

- **背景**：实录与纲目都很耗时，但纲目其实是**基于转写原文独立归纳**的（带时间戳时直接读原稿、不读实录内容，见 [v0.5.1 §2.2](开发文档_v0.5.1.md)）。此前 `chapters.requires=["transcript"]` 这道门禁会逼用户先等实录跑完，纯属多余等待。
- **改动**：
  - `modules/business_school/config.py`：纲目 `StepDef` 去掉 `requires=["transcript"]`。
  - `pipeline/engine.py` `_step_chapters`：增加空输入保护——纲目可能先于实录运行，未上传转写原文时给出明确报错而非空跑。
  - 前端无需改动：纲目 Tab 灰态动态读 `step.requires`（`Workbench.checkDeps`），后端去掉后自动解锁（仍要求已上传素材）。
- **效果**：实录、纲目是不同 step，runner 本就按 `(sid, step)` 各自并发——解除门禁后即可**并行触发**，互不阻塞。

## 3. 产出依赖关系一览（v0.5.2）

| 产出 | 执行前置（门禁） | 内容素材来源 |
|------|------------------|--------------|
| 实录 transcript | 无（需已上传素材） | 转写原文 |
| 纲目 chapters | **无**（需已上传素材） | 转写原文（带时间戳时）/ 无时间戳时退用实录文本 |
| 撷要 minutes_concise | 实录 | 实录正文（+纲目可选参考） |
| 笺注 minutes_detailed | 实录 | 实录正文（+纲目可选参考） |
| 脉络 graph | 撷要 或 笺注 | 纪要正文 |

> 可并行：实录 ∥ 纲目（二者皆只依赖原文）。撷要/笺注须待实录，脉络须待撷要/笺注。

## 4. P5 智能问答（课堂答疑）—— 规划

**目标**：工作台提供独立的「提问」窗口，用户就本堂课自由发问，AI 依据**该会话的素材（原始转写稿 / 辅助资料）+ 已生成产出（实录、纲目、撷要、笺注、脉络）**作答，做到「有据可查、不杜撰」。

**排期决策**：安排在 **P3 辅助素材之后**。理由——问答要「根据提供的资料回答」，而当前后端只消化 txt/md，PDF/图片素材要等 P3 接入后才能真正进入问答上下文；产出侧已具备，可直接引用。P3 完成即解锁高质量问答的数据底座，P5 紧随其后。在 P3 之前若要先做，可先支持「仅基于已有产出 + txt/md 素材」的简化版。

**实现要点（拟）**：
- 后端：新增 `POST /api/sessions/{id}/ask`（body `{question, history?}`，SSE 流式返回）。上下文 = 选取相关产出 + 素材摘要拼装进 system/user；超长按需检索/截断（先简单拼接，后续可加轻量检索）。问答历史落盘 `data/sessions/<id>/qa/`，刷新可续。
- Provider：复用现有模型层与会话级模型覆盖（问答可单独选模型）；安全红线沿用「不杜撰数字/人名/公司名，答不到就说没有」。
- 前端：工作台新增「问答」Tab 或侧边抽屉，聊天式 UI，复用 `attachStream` 接 SSE；引用来源可标注「出自：笺注 / 纲目 阶段3」。
- 提示词：新增 `qa_system/qa_user`，强调「只依据给定资料与产出回答，超出范围明确说明」。

## 5. 下一步候选

1. **P3 辅助素材**（建议优先）：前端已能上传 pdf/图片，但后端只读 txt/md——属半成品。计划：PDF 用 `pypdf` 抽文本、图片走视觉模型（Provider 已有 `supports_vision`），作为背景注入各产出提示词。这也是 P5 问答的前置数据底座。
2. **导出**：产出导出 md / docx / pdf，脉络图导出 PNG/SVG；可整场打包 zip。
3. **P5 智能问答**：见 §4，排在 P3 之后。
4. **生成用时记录**（小功能，轻量）：记录并展示每次产出生成耗时，便于评估速度优化效果、对比不同模型快慢。
   - 后端：在 runner/engine 的 `start`/`done` 事件里带上时间戳（已有 `t` 字段），或在 `done` 事件附 `elapsed` 秒数；可顺带把"本次用时 + 使用的 provider/model"写入版本 note 或产出元数据，留作历史。
   - 前端：产出 Tab 或面板头显示「本次生成 12.3s」；查看历史版本时也能看到当时用时。
   - 进阶（可选）：分块/分段任务可记录"墙钟用时 vs 累计 token"，辅助判断并行收益与限流瓶颈。
5. **实录分块截断保护**（健壮性，配合大块）：当前实录块 10000 字、输出上限 16000 token，按 deepseek-v4-flash（384K 输出）调的安全值；但用户可手动换成单次输出上限较小的模型，等长改写会【静默截断】（结尾莫名其妙断掉）。
   - 计划：调用返回 `finish_reason == "length"`（或输出长度贴近 `max_tokens`）时，自动把该块二分重切重试，直到不再截断——这样可放心用大块，又不怕偶发截断。
   - 配套：超长稿（块数 > 并发数）时适度提高 `MAX_PARALLEL`（而非缩小块），把长稿压到尽量少的并行轮次。
6. **P4**：术语库自进化、其余板块（矿业有 `sichuan-mining-transcribe` skill）、设置页。

（实录/纲目内容生成机制见 [开发文档_v0.5.1.md](开发文档_v0.5.1.md)；P2 修订机制见 [开发文档_v0.4.0.md](开发文档_v0.4.0.md)。）

## 6. 已知小问题

- Markdown 渲染未消毒（`dangerouslySetInnerHTML`）——本地单用户风险低，上云/多用户阶段加 DOMPurify。
- 实录角色标注依赖模型语义判断：时间戳已确定性清除；发言人标签在"模型成功但未遵守"时仍可能偶现（空返回/失败的块已由兜底彻底剥离）。
- 纲目分段在段边界可能把同一主题切成相邻两个阶段（细颗粒度的代价，符合 v1 风格）。
- 删除「当前默认」供应商时默认回退到列表第一个，建议删前先改默认。
