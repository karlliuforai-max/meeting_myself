# 会议纪要生成平台 · 开发文档 v0.6.2

> 历史版本：[v0.6.0](开发文档_v0.6.0.md) · [v0.6.1](开发文档_v0.6.1.md)。

**v0.6.2 状态**：修复两个回归/质量问题：老会话历史版本在新产出名下不可见；纲目最终汇编可能被截断但仍被当作成功结果落盘。

## 1. 修复一：历史版本跨文件名合并

### 问题

早期版本产出名曾从「逐字稿 / 章节稿 / 纪要主体 / 知识图谱」调整为「实录 / 纲目 / 撷要 / 笺注 / 脉络」。内容读取已有 `LEGACY_NAMES` 兼容，但版本列表仍只查新文件名目录。

因此老会话会表现为：当前产出能看到，但旧历史版本不显示；重新生成后只看到新文件名目录里的版本。

### 修复

API 层新增版本兼容映射：

- `GET /artifacts/{name}` 返回旧名版本 + 新名版本合并后的连续版本线。
- `GET /artifacts/{name}/versions/{version}`、历史版本下载、恢复版本均按连续版本号映射回真实磁盘文件。
- 不迁移用户数据，不改动已有版本目录。

## 2. 修复二：纲目完整性闸门

### 问题

“中美科技核心差异”项目里，旧代码生成出只有 20 阶段且末尾为 `### 阶段20：` 的残缺纲目。根因是最终汇编结果虽然低于本堂课预算区间（目标约 30，允许 26-34），但仍落在全局硬边界 20-50 内，旧兜底逻辑把它当成成功结果。

### 修复

- 最终汇编输出上限 `CHAPTERS_FINAL_MAX_TOKENS` 从 7000 提高到 12000。
- 新增 `_chapters_complete()`：拦截空结果、无阶段、空阶段标题等明显截断结果。
- 兜底策略收紧：低于本堂课预算下界的结果不再被放行；若汇编/修复结果不达标，则优先回退完整候选纲目，否则报错，不写坏当前产出。
