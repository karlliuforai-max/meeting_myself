## 1. 项目目标

本项目是一个本地优先的多场景会议纪要生成平台。用户会将录音转写稿、课堂笔记、辅助资料等输入到网页工作台，由平台按不同场景生成结构化的个人知识产物。

当前优先场景是「学堂 · 商学院课堂讲座」：

- 主体素材：录音转写后的 `txt/md` 文本。
- 辅助素材：课堂笔记照片、板书照片、讲义照片。
- 暂缓素材：PDF 辅助资料，等有真实测试样例后接入。
- 产出目标：既能快速回看课堂，也能沉淀完整知识结构和逻辑图谱。

后续预留场景：

| 板块 | 状态 | 说明 |
|------|------|------|
| 学堂 · 商学院课堂讲座 | 已开放 | 普通话/英语为主的课堂、讲座、分享 |
| 矿山 · 矿业商务沟通 | 占位 | 四川话/乐山话夹普通话，未来可接入既有转写 skill |
| 闲谈 · 日常聊天 | 占位 | 高价值日常交流整理 |
| 通用 | 占位 | 未归类会议纪要 |

## 3. 当前系统架构

### 3.1 后端

后端使用 FastAPI，核心目录如下：

```text
backend/
  main.py                 FastAPI 入口，生产模式托管 frontend/dist
  config.py               配置与 VERSION 读取
  api/routes.py           API 路由层
  providers/              模型 Provider 抽象与动态配置
  modules/                场景板块定义
  pipeline/               生成流水线、后台任务、转写/视觉处理
  storage/                本地会话存储与版本管理
  tests/                  unittest 测试
```

后端当前有三个关键抽象：

- **ModuleDef / StepDef**：定义场景板块、产出步骤、依赖关系、默认模型。
- **Provider**：抽象 Claude、DeepSeek、自定义 OpenAI 兼容端点、Anthropic Messages 兼容端点。
- **SessionStore**：本地会话、输入文件、当前产出、历史版本、进度事件的存储层。

### 3.2 前端

前端使用 React + Vite，核心目录如下：

```text
frontend/src/
  App.jsx                 顶层布局、板块切换、模型配置入口
  api.js                  后端 API 封装
  pages/ModuleHome.jsx    板块首页、项目列表、新建项目
  pages/Workbench.jsx     工作台，上传、背景、生成、修订、导出
  components/ModelConfig.jsx
  components/OutputView.jsx
  components/InlineEdit.jsx
  styles.css              全局样式
```

生产模式下前端先构建到 `frontend/dist/`，后端直接托管静态文件。用户只需启动后端即可访问网页。

## 4. 会话与文件结构

一个会话对应一堂课或一次会议，落盘在：

```text
data/sessions/<sid>/
  meta.json
  inputs/
  artifacts/
  versions/
  progress/
  derived/
```

主要文件含义：

| 路径 | 内容 |
|------|------|
| `meta.json` | 会话标题、板块、补充背景、各步骤模型选择、状态 |
| `inputs/` | 用户上传的原始素材，包含转写文本、图片、未来 PDF |
| `artifacts/` | 当前版本产出 |
| `versions/<产出名>/vN.md` | 某产出的历史版本 |
| `versions/<产出名>/vN.note.txt` | 版本说明，例如“生成”“修订：...” |
| `progress/<step>.json` | 单步骤 SSE 进度历史 |
| `derived/notes/` | 图片笔记视觉识别缓存 |

`data/` 被 Git 忽略，不提交原始转写稿、图片、API Key 或用户私有数据。

## 5. 学堂板块产出设计

当前商学院课堂讲座板块有五类产出：

| 产出 | 文件名 | 输入依赖 | 默认模型 | 定位 |
|------|--------|----------|----------|------|
| 实录 | `实录.md` | 转写原文 | `deepseek-v4-flash` | 纠错后的逐字稿，去时间戳和口水词，保留课堂讲话结构 |
| 纲目 | `纲目.md` | 转写原文，可独立生成 | `deepseek-v4-flash` | 按课堂时间顺序提炼 20-50 个阶段 |
| 撷要 | `撷要.md` | 实录，可选纲目、笔记照片 | `claude-sonnet-4-6` | 精炼纪要，便于快速回看 |
| 笺注 | `笺注.md` | 实录，可选纲目、笔记照片 | `claude-sonnet-4-6` | 详尽纪要，便于深度学习 |
| 脉络 | `脉络.mmd` | 撷要或笺注 | `claude-sonnet-4-6` | Mermaid 逻辑图谱 |

依赖关系：

```text
转写原文 ──► 实录 ──┬──► 撷要 ──┐
          └──► 纲目 ─┘          ├──► 脉络
课堂笔记图片 ───────► 撷要/笺注 ─┘
```

说明：

- 实录和纲目都可以基于转写原文独立生成，因此可以并行触发。
- 撷要和笺注依赖实录，若已有纲目会一并参考。
- 课堂笔记照片只注入撷要/笺注，不混入实录，也暂不进入纲目。
- 脉络优先基于笺注，没有笺注时可基于撷要。

## 6. 图片笔记视觉链路

v0.6.0 已完成图片笔记识别：

```text
图片输入
  -> Pillow 降采样到最长边 1568px
  -> base64
  -> 视觉模型多模态 chat
  -> Markdown 笔记转录
  -> derived/notes/ 缓存
  -> 注入撷要/笺注提示词
```

识别规则：

- 手写、板书、PPT、讲义都按结构化 Markdown 转录。
- 公式转 LaTeX，表格转 Markdown。
- 字迹不清标 `[字迹不清]`，拿不准标 `[存疑]`。
- 不杜撰，不把笔记内容当成课堂原话。
- 与逐字稿冲突时，以逐字稿为准。

模型选择：

1. 用户在模型配置面板手动指定“图片识别模型”。
2. 若未指定，自动选择已配置且支持视觉的供应商，优先全局默认供应商。
3. 若没有可用视觉模型，则跳过图片，并在进度中提示。

## 7. 模型配置与调用

模型配置从静态 `.env` 演进为运行时可视化管理。

### 7.1 Provider 配置

界面入口：右上角“模型配置”。

支持：

- 新增、编辑、删除供应商。
- 设置接口协议：OpenAI Chat Completions 或 Anthropic Messages。
- 配置 Base URL、API Key、模型列表、默认模型。
- 标记是否支持视觉。
- 设为全局默认。
- 连通性测试，支持测试未保存草稿。

配置持久化到：

```text
data/providers.json
```

`.env` 只在首次初始化且 `providers.json` 不存在时用于播种默认供应商。之后日常修改以界面配置为准。

### 7.2 分步默认模型

每个产出都有“首选默认模型名”，但不写死供应商 ID：

- 实录、纲目优先 `deepseek-v4-flash`。
- 撷要、笺注、脉络优先 `claude-sonnet-4-6`。

解析逻辑：

1. 如果用户为某个产出手动选择了 provider/model，优先使用手动配置。
2. 否则在已配置供应商中寻找包含该默认模型名的供应商。
3. 找不到则回退全局默认供应商及其默认模型。

## 8. 生成与进度机制

### 8.1 单步生成

每个产出独立触发：

```http
POST /api/sessions/{sid}/run-step?step=chapters
```

后台任务维度是 `(sid, step)`，同一个会话的不同步骤可以并行，同一步骤不能重复跑。

### 8.2 SSE 进度

前端通过：

```http
GET /api/sessions/{sid}/run-step-stream?step=chapters
```

订阅进度事件。事件也会落盘到 `progress/<step>.json`，刷新页面后可恢复 UI 状态。

### 8.3 持续修订

每个产出生成后可以继续填写修订意见：

```http
POST /api/sessions/{sid}/revise-step?step=minutes_detailed
```

修订逻辑：

- 输入：当前版本内容 + 用户修订意见。
- 输出：完整的新版本。
- 历史版本继续保留。
- 支持恢复任意历史版本为当前版本，恢复也会写成一个新版本。

## 9. 导出能力

v0.6.1 已实现导出 MVP。

### 9.1 单个产出下载

```http
GET /api/sessions/{sid}/artifacts/{name}/download
```

特点：

- 下载当前产出。
- 支持中文文件名。
- `md` 使用 `text/markdown`，`mmd` 使用文本下载。
- 兼容老会话旧产出名。

### 9.2 历史版本下载

```http
GET /api/sessions/{sid}/artifacts/{name}/versions/{version}/download
```

前端查看历史版本时，“下载当前”会切换为“下载此版本”。

### 9.3 项目产出 ZIP

```http
GET /api/sessions/{sid}/exports/bundle
```

ZIP 内容只包含已生成产出，不包含原始输入素材。这样可以避免把转写稿、课堂笔记图片等私有数据无意打包出去。

## 10. v0.6.2 修复一：历史版本跨文件名合并

### 10.1 问题现象

用户反馈：之前可以看到每一个历史版本，现在重新生成后似乎只能看到新版本。

### 10.2 根因

项目早期改过产出文件名：

| 旧名 | 新名 |
|------|------|
| `逐字稿.md` | `实录.md` |
| `章节稿.md` | `纲目.md` |
| `纪要-精炼版.md` / `纪要主体.md` | `撷要.md` |
| `纪要-详尽版.md` | `笺注.md` |
| `知识图谱.mmd` | `脉络.mmd` |

当前产出读取已经支持 `LEGACY_NAMES`，所以老内容能显示。但版本列表只查询新文件名目录，例如只查：

```text
versions/实录.md/
```

不会查旧目录：

```text
versions/逐字稿.md/
```

因此旧版本没有丢，只是没有被 API 合并返回。

### 10.3 修复方案

在 `backend/api/routes.py` 增加版本兼容层：

- `_version_source_names(name)`：返回旧文件名列表 + 新文件名。
- `_version_entries_compat(sid, name)`：把多个真实版本目录合并为连续版本号。
- `_read_version_compat(sid, name, version)`：将前端看到的连续版本号映射回真实磁盘文件。

受影响接口：

```http
GET  /api/sessions/{sid}/artifacts/{name}
GET  /api/sessions/{sid}/artifacts/{name}/versions/{version}
GET  /api/sessions/{sid}/artifacts/{name}/versions/{version}/download
POST /api/sessions/{sid}/artifacts/{name}/versions/{version}/restore
```

修复原则：

- 不迁移用户数据。
- 不重写历史版本目录。
- 前端无需知道旧文件名。
- 连续版本号以旧版本在前、新版本在后的顺序展示。

## 11. v0.6.2 修复二：纲目完整性闸门

### 11.1 问题现象

“中美科技核心差异”项目的纲目曾生成不完整：

- 旧结果只有 20 个阶段。
- 末尾出现空标题：`### 阶段20：`。
- 进度里显示本堂课按时长规划约 30 个阶段，允许范围约 26-34。

### 11.2 根因

纲目生成分两层：

1. 分段并行生成候选阶段。
2. 全局汇编，合并重复或碎片阶段，得到最终纲目。

旧逻辑的问题在最终兜底：

- 全局硬边界是 20-50 个阶段。
- 但每堂课还有动态目标区间，例如目标 30，允许 26-34。
- 当模型最终汇编被截断，只返回 20 个阶段时，虽然低于本堂课动态下界，但仍满足全局硬边界 20-50。
- 旧代码因此把残缺结果当作成功产出落盘。

### 11.3 修复方案

修改 `backend/pipeline/engine.py`：

- `CHAPTERS_FINAL_MAX_TOKENS` 从 7000 提高到 12000，降低长纲目被截断的概率。
- 新增 `_chapters_complete(md)`。
- 拦截以下明显不完整结果：
  - 空内容。
  - 没有任何阶段标题。
  - 末尾或任意位置出现空阶段标题，例如 `### 阶段20：`。
- 收紧兜底策略：
  - 符合本堂课动态范围的完整汇编结果优先。
  - 修复汇编仍不达标时，可回退完整候选纲目。
  - 只允许“偏多但完整”的宽容兜底，不允许低于动态下界的结果写入。
  - 如果所有结果都不完整或低于下界，则抛出 `ProviderError`，前端显示错误，不覆盖当前产出。

### 11.4 已处理数据

已用新代码重新生成“中美科技核心差异”的纲目：

- 当前最新版本：合并版本线 `v4`。
- 阶段数：26 个。
- 时间覆盖：从开场到 `03:01:20`。
- 已进入本堂课动态范围下界，替代旧的残缺结果。

## 15. 已知取舍

1. PDF 暂未接入生成
   - 前端已允许上传 PDF。
   - 后端依赖已有 `pypdf`。
   - 由于当前真实测试数据主要是转写文本 + 图片笔记，PDF 抽文本顺延到有样例后做。

2. 导出仍是 MVP
   - 当前导出为 Markdown/Mermaid 源文件级导出。
   - 尚未做合并 Markdown、DOCX、PDF、脉络图 SVG/PNG。

3. 纲目完整性检查仍是结构性防护
   - 已能拦截明显截断结果。
   - 不能保证模型每个阶段的内容质量都完美，质量仍依赖所选模型和提示词。

4. Markdown 渲染未加 DOMPurify
   - 当前定位是本地单用户，风险较低。
   - 若未来上云或多人使用，应加入 HTML 消毒。

5. 生成进度仍有“模型调用期间长时间停在某一步”的体验问题
   - SSE 有历史回放和 ping。
   - 但纲目内部并行分段到最终汇编之间，中间细粒度进度还不够丰富。

## 16. 下一步建议

按当前真实使用价值排序：

1. **导出增强一：合并 Markdown**
   - 把实录、纲目、撷要、笺注、脉络源码合并为一份完整课堂文档。
   - 作为 DOCX/PDF 的前置基础。

2. **导出增强二：DOCX / PDF**
   - 面向正式归档、打印、分享。
   - 需要定义标题页、目录、分节、样式、页眉页脚。

3. **脉络图导出 SVG / PNG**
   - 当前可下载 `.mmd` 源码。
   - 后续可把前端渲染后的 SVG 或 PNG 一并导出。

4. **PDF 抽文本**
   - 在用户提供真实 PDF 样例后接入。
   - 建议与图片一样作为辅助素材，优先注入撷要/笺注。

5. **P5 智能问答**
   - 建立“根据输入素材 + 五类产出答疑”的聊天窗口。
   - 需要处理上下文选择、引用来源、问答历史落盘。

6. **工程小项**
   - 生成耗时记录。
   - Provider 返回截断原因识别。
   - Mermaid chunk 拆分或按需加载。
   - DOMPurify。
